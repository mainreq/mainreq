MainReq: Definiciones y Preguntas Frecuentes


Elementos del Proyecto:

En esta sección puedes editar (si el administrador te ha otorgado permisos de edición) y revisar los elementos del proyecto. Éstos son Hitos, Tipos de Usuario, Requisitos de Usuario, Requisitos de Software, Módulos y Casos de Prueba.

Puedes ordenarlos por sus atributos principales y expandir el contenido para revisar búsquedas rápidas (usando "ctrl+f" en el navegador).

Puedes revisar el historial de cambios de cada elemento en la Bitácora.

Las Alarmas (icono azul "( ! )") te muestran posibles conflictos en la especificación de cada elemento.

Puedes descargar un documento PDF con el listado y detalle de los Elementos del Proyecto.


Hitos: HT

  • Es un elemento del proyecto...
  • Definición: El proyecto se puede dividir en incrementos, los que se identifican con un Hito de revisión o entrega de avance. Un Hito Consta de:
    • Identificador: Código único que sirve para identificar y referenciar formalmente este elemento dentro del proyecto. Es asignado automáticamente de manera incremental utilizando el formato "HTXXXX".
    • Nombre: Nombre en lenguaje normal del hito.
    • Descripción: Descripción del período marcado por este hito.
    • Fechas de inicio y fin: Las fechas (con el formato "AAAA-MM-DD hh:mm") de inicio y fin que contemplan la etapa de desarrollo definida por este Hito.
  • Alarmas:
    • "Este Hito no ha sido asignado a ningún Requisito de Usuario"
    • "Este Hito no ha sido asignado a ningún Requisito de Software"

Tipos de Usuario: TU

  • Es un elemento del proyecto...
  • Definición: El producto de software a desarrollar puede tener distintos Tipos de Usuario que interactúan con él. Un Tipo de Usuario consta de:
    • Identificador: Código único que sirve para identificar y referenciar formalmente este elemento dentro del proyecto. Es asignado automáticamente de manera incremental utilizando el formato "TUXXXX".
    • Nombre: Nombre en lenguaje normal del Tipo de Usuario.
    • Descripción: Acá se describen las características de este Tipo de Usuario.
    • Cantidad: Número entero positivo correspondiente a una estimación de la cantidad de usuarios del sistema que corresponden a este tipo.
    • Usuarios Localizables: Listado de nombre y datos de contacto (ej.: correo electrónico y teléfono) de usuarios conocidos y localizables durante el desarrollo del proyecto. Los datos de cada uno pueden ser separados por un punto y coma ";".
  • Alarmas:
    • "Este Tipo de Usuario no ha sido asociado a ningún Requisito de Usuario"
    • "Este Tipo de Usuario no ha sido asociado a ningún Requisito de Software"
    • "Este Tipo de Usuario no ha sido asociado a ningún Caso de Prueba"

Requisitos de Usuario: RU

  • Es un elemento del proyecto...
  • Definición: Corresponden a los requisitos acordados con el cliente, y describen el sistema a desarrollar. Se recomienda que cumplan con las siguientes características:
    • Necesario: Un requisito es necesario si su omisión provoca una deficiencia en el sistema a construir.
    • Conciso: Un requisito es conciso si es fácil de leer y entender. Su redacción debe ser simple y clara para aquellos que lo consulten en el futuro.
    • Completo: Un requisito está completo si no necesita ampliar detalles en su redacción, es decir, si muestra información suficiente para su comprensión.
    • Consistente: Un requisito es consistente si no se contradice con otro requisito.
    • No ambiguo: Cuando el lenguaje usado en su definición no causa confusiones al lector y tiene una única interpretación.
    • Verificable: Un requisito es verificable cuando puede ser cuantificado de manera que permita hacer uso de algún método de verificación.
  • Un Requisito de Usuario consta de:
    • Identificador: Código único que sirve para identificar y referenciar formalmente este requisito dentro del proyecto. Es asignado automáticamente de manera incremental utilizando el formato "RUXXXX".
    • Nombre: Nombre en lenguaje normal del requisito.
    • Descripción: Descripción del requisito (los aspectos que involucra, en qué consiste, etc.).
    • Urgencia: Nivel de urgencia asociada al requisito. La urgencia de un requisito puede ser "urgente", "normal" o "si se puede".
    • Estabilidad: Este atributo determina si el requisito puede o no estar sujeto a cambios durante el ciclo de vida del producto de software. La estabilidad puede ser "transable" o "intransable".
    • Tipo:
      • Funcional: Describe una capacidad que debe ser soportada por el producto de software.
      • De Calidad: Representan un atributo de calidad del producto, y se pueden expresar en una escala de medición.
      • De Restricción: Indican con qué restricciones se desarrollará y/o funcionará el software, en función de costos, tiempos, personal, ambiente operacional, hardware, redes, etc.
    • Fuente: Documento o persona desde la cual surge el requisito.
    • Estado: El estado de cumplimiento actual del requisito dentro del desarrollo, puede ser "Cumple", "No Cumple" o "Ambiguo". El estado predeterminado al momento de la creación es "No Cumple".
    • Costo: Número entero positivo que representa el costo asociado al desarrollo de este requisito, útil en caso de utilizar "Puntos de Función" como método de estimación de costos.
    • Hito: Asociado a un Hito particular del Proyecto.
    • Tipos de Usuario: Usuarios involucrados con este requisito.
  • Alarmas:
    • "El Hito asignado a este Requisito de Usuario ha sido eliminado"
    • "No tiene ningún Requisito de Software asociado"
    • "Tiene más de 5 Requisitos de Software asociados (puede ser muy complejo y quizás deba ser dividido)"
    • "Su urgencia es menor que la urgencia más baja de sus Requisitos de Software asociados"
    • "Su estabilidad es intransable, sin embargo todos sus Requisitos de Software asociados son transables"
    • "Uno de sus Tipos de Usuario no se encuentra asociado a ninguno de sus Requisitos de Software"
    • "Su estado es No Cumple, pero todos sus Requisitos de Software asociados Cumplen"
    • "Su estado es No Cumple, pero todos sus Casos de Prueba asociados Cumplen"
    • "Su estado es Cumple, pero ninguno de sus Requisitos de Software asociados Cumplen"
    • "Su estado es Cumple, pero ninguno de sus Casos de Prueba asociados Cumplen"

Requisitos de Software: RS

  • Es un elemento del proyecto...
  • Definición: Son los requisitos que especifica lo que necesita el software a desarrollar para satisfacer los requisitos de usuario. Sus atributos son similares a los de los Requisitos de Usuario, salvo:
    • El formato del identificador asignado automáticamente es "RSXXXX".
    • Se asocia con los requisitos de usuario correspondientes mediante una lista de selección múltiple.
    • Los tipos de requisitos son más específicos:
      • Funcionales: Indican cuáles deben ser las capacidades del software. Se derivan del modelo lógico.
      • Interacción: Especifican el hardware, software o elementos de bases de datos con los que el sistema o sus componentes interactúan o se comunican.
      • Operacionales: Especifican la forma en que correrá el sistema y cómo se comunicará con los operadores humanos. Incluyen todas las interfaces de usuario, interacción humano-computador, y requisitos logísticos y organizacionales.
      • Recursos (Ambiente Operacional): Especifican los límites superiores de los recursos físicos, tales como capacidad de procesamiento, uso de memoria, espacio en disco, etc.
      • Usabilidad: Están relacionados con el esfuerzo de uso y la evaluación del uso realizada por los usuarios.
      • Mantenibilidad: Relacionados con el esfuerzo de hacer modificaciones, especificando cuán fácil es la reparación de fallas y adaptación del software a nuevos requisitos.
      • Portabilidad: Relacionados con la habilidad del sistema de ser transferido de un ambiente a otro.
      • Confiabilidad: Están relacionados con la capacidad de mantener un nivel adecuado de servicio, bajo ciertas condiciones y por cierto tiempo. Especificando los tiempos medios entre fallas aceptables.
      • Interoperabilidad: Habilidad de interactuar con determinados sistemas.
      • Rendimiento: Establecen valores numéricos para variables medibles relacionadas con el rendimiento del sistema.
      • Documentación: Especifican requisitos particulares del proyecto para la documentación.
      • Escalabilidad: Especifica la capacidad del sistema para mantener o mejorar su rendimiento medio conforme al aumento del número de usuarios.
  • Alarmas:
    • "El Hito asignado a este Requisito de Software ha sido eliminado"
    • "Su Hito no corresponde con ningún Hito asociado a sus Requisitos de Usuario"
    • "No tiene ningún Requisito de Usuario asociado"
    • "Tiene más de 3 Requisitos de Usuario asociados (puede ser muy complejo y quizás deba ser dividido)"
    • "Su urgencia es mayor que la urgencia más alta de sus Requisitos de Usuario asociados"
    • "Su estabilidad es intransable, sin embargo todos sus Requisitos de Usuario son transables"
    • "Un Requisito de Usuario asociado ha sido modificado (su fecha de modificación es posterior a la de este Requisito de Software, favor revisar y actualizar)"
    • "Uno de sus Tipos de Usuario no se encuentra asociado a ninguno de sus Requisitos de Usuario"
    • "No tiene ningún Módulo asociado"
    • "Su estado es No Cumple, pero todos sus Casos de Prueba asociados Cumplen"
    • "Su estado es Cumple, pero ninguno de sus Casos de Prueba asociados Cumplen"

Módulos: MD

  • Es un elemento del proyecto...
  • Definición: Corresponden a componentes de alto nivel que encapsulan los requisitos de software descritos en etapas anteriores. En la práctica éstos se pueden definir como interfaces, páginas web o procedimientos. Un Módulo se compone de:
    • Identificador: Código único que sirve para identificar y referenciar formalmente este módulo dentro del proyecto. Es asignado automáticamente de manera incremental utilizando el formato "MDXXXX".
    • Nombre: Nombre simple del módulo.
    • Descripción: Se describe la función que encapsula este módulo.
    • Urgencia: Nivel de urgencia asociada al módulo. La urgencia de un módulo puede ser "urgente", "normal" o "si se puede". El formulario ofrece una lista seleccionable.
    • Costo: Se ingresa de la misma que el costo asociado a los requisitos, corresponde a un número entero positivo que representa el costo asociado al desarrollo de este módulo.
    • Requisitos de Software: sub-componentes que integran el Módulo.
  • Alarmas:
    • "No tiene ningún Requisito de Software asociado"
    • "Ningún Requisito de Software asociado tiene por lo menos la misma urgencia que este módulo"
    • "Un Requisito de Software asociado ha sido modificado (su fecha de modificación es posterior a la de este Módulo, favor revisar y actualizar)"

Casos de Prueba: CP

  • Es un elemento del proyecto...
  • Definición: son usados para detectar errores en el sistema desarrollado y como medio para validar que el producto de software ha sido concebido e interpretado correctamente según lo especificado en los requisitos. Un Caso de Prueba consta de:
    • Identificador: Código único que sirve para identificar y referenciar formalmente este caso de prueba dentro del proyecto. Es asignado automáticamente de manera incremental utilizando el formato "CPXXXX".
    • Nombre: Nombre simple del caso de prueba.
    • Descripción: Acá se describe el procedimiento para evaluar el caso de prueba.
    • Resultado Aceptable: Se describe el mínimo resultado aceptable que hace exitosa a esta prueba.
    • Resultado Óptimo: Se describe el mejor resultado esperable tras realizar la prueba.
    • Estado: El estado de cumplimiento actual del caso de prueba, puede ser "Cumple", "No Cumple" o "Ambiguo". El estado predeterminado al momento de la creación es "No Cumple".
    • Requisito: Un Caso de Prueba debe estar asociado a un Requisito de Software.
    • Tipos de Usuario: Se deben especificar los usuarios que tienen participación en las pruebas a realizar.
  • Alarmas:
    • "El Requisito asignado a este Caso de Prueba ha sido eliminado"
    • "El Requisito asociado ha sido modificado (su fecha de modificación es posterior a la de este Caso de Prueba, favor revisar y actualizar)"

Documentos del Proyecto:

La sección de "Documentos" sirve para editar y descargar los siguientes documentos: Documento de Requisitos, Documento de Diseño, Documento de Casos de Prueba y el Documento Histórico del Proyecto.

El editor de texto enriquecido incorpora herramientas básicas, permitiendo redactar párrafos y subir imágenes. Además se puede revisar versiones anteriores de cada sección del documento (descuida, sólo se almacena una nueva versión al guardar los cambios).

Preguntas Frecuentes:

  • ¿Cómo hacer que una imagen quede bien centrada en el documento PDF?
    • Se debe centrar el texto/cursor (con el botón del editor) y luego insertar la imagen ahí. Si se inserta la imagen y luego se centra no funcionará al generar el documento PDF.
  • ¿Por qué no puedo usar "h1" ni "h2" para dar formato de títulos en la edición de documentos?
    • Porque son usados para darle forma al documento final.
  • ¿Qué pasa si una imagen es muy grande para una página del documento PDF?
    • Automáticamente se ajusta la imagen al tamaño máximo visible en la hoja.
  • ¿Por qué cuando me cambio de documento las secciones no cambian?
    • Porque en realidad a los Documentos de Requisitos, Casos de Prueba y Diseño les corresponden sólo algunas de las secciones compartidas presentes en el Documento Histórico.

Breve descripción de las Secciones de un Documento:

  • Equipo Desarrollador y Contraparte: Acá se deben listar el nombre, rol y contacto (email), de los integrantes pertenecientes al equipo de desarrollo y a la contraparte.
  • Acceso al Prototipo del Proyecto: Acá se deben colocar las URL y los usuarios/contraseñas (si corresponde) necesarias para probar la versión actual del software desarrollado.
  • Introducción: En esta introducción se describe brevemente el contexto, objetivos y alcance del sistema a desarrollar, así como la documentación relativa al mismo.
  • Propósito: Describir aquí qué hace y qué es el sistema. Para proyectos pequeños, media página debería ser más que suficiente.
  • Alcance: Describir hasta dónde llega el proyecto: qué es y qué no es. Para proyectos pequeños, media página debería ser más que suficiente.
  • Contexto: Dar información respecto del contexto del desarrollo y el contexto en el que se tiene que insertar el sistema. Tecnologías que estarán involucradas, trabajos previos, vínculos con otros sistemas, etc. Escriba lo que necesite, en general los gráficos son bienvenidos, pues ayudan mucho a la comprensión.
  • Definiciones: Indique aquí la definición de las "palabras claves" (o keywords) que se emplearán en el documento, y que el lector no necesariamente conoce. De la misma manera, coloque el significado de todas siglas o abreviaturas que se empleen. Tanto las siglas como las definiciones tienen que expresar lo que el/los autores del documento entienden, y no necesariamente dar definiciones generales. La idea es poder entender en toda su magnitud el documento que se está leyendo, y nada más. La lista DEBE estar en orden alfabético para facilitar la búsqueda de conceptos.
  • Referencias: Enumere la documentación y bibliografía utilizada como apoyo para construir este documento. Coloque fechas y versiones de documentos cuando corresponda.
  • Descripción General: Esta sección describe los requisitos funcionales de los Usuarios/Clientes, sus interfaces externa, las condiciones de excepción y las clases de pruebas que se harán para verificar que los requisitos se cumplen.
  • Usuarios: Acá se deben identificar los tipos de usuarios, los atributos generales de cada tipo y la cantidad de personas en cada categoría.
  • Producto: Acá debe colocarse la perspectiva que tienen los clientes y los usuarios acerca del producto. Cada tipo de usuario tiene diferentes expectativas, y cada tipo de cliente también. El cliente es el que aporta la visión de la organización. Eso debería estar expresado acá. Basta con colocar una oración por cada tipo de usuario y cliente.
  • Ambiente: Acá se describen brevemente los componentes principales del ambiente operacional, donde deberá vivir la solución que está siendo desarrollada. Esto puede describir el escenario actual, o bien el futuro, si es que se realizará una compra de equipamiento. La descripción se realiza a nivel de hardware, servidores, bases de datos, redes, etc.
  • Proyectos Relacionados: Acá se debe especificar si este proyecto tiene relación con algún sistema ya implementado, con otro proyecto en ejecución o planificado.

Tareas

Figura 33: Ciclo de vida de Tareas.

  • En esta sección un integrante del equipo de desarrollo puede ver y reportar el estado de las Tareas que se le han encargado. Si el usuario tiene permisos de edición (como un "Jefe de Proyecto"), podrá asignar y revisar Tareas.
  • La Figura 33 muestra los posibles estados de una Tarea.
    • Por Hacer: La Tarea ha sido creada.
    • Haciendo: El integrante se encuentra haciendo la tarea.
    • Realizada: El integrante responsable reporta la realización de la Tarea para que sea evaluada.
    • Aprobada: El evaluador considera válida la Tarea reportada por el integrante.
    • Reprobada: El evaluador considera inválida/insuficuente la Tarea reportada por el integrante.
    • Descartada: El evaluador decide que la especificación de la Tarea es inconsistente o ambigua.
  • Si vence el plazo de entrega, la tarea se marca como atrasada.

Estadísticas del Proyecto

En esta sección se puede revisar el avance del proyecto en términos de cumplimiento y cantidad de elementos abarcados por Hito.

Se ofrece un selector para revisar las estadísticas de elementos asociados a un Hito particular del proyecto. Las tablas pueden ser descargadas en un documento PDF con portada, índice y separadas por hito para su posterior revisión.

  • Requisitos de Usuario: Se despliega la cantidad separada por atributos (urgencia, estabilidad y tipo) y por el nivel de cumplimiento (cumple, no cumple y ambiguo). Contabilizando el costo total y cuántos de ellos no tienen Requisito de Software o Tipo de Usuario asociados.
  • Requisitos de Software: Muestra la misma información que la tabla de Requisitos de Usuario, pero considerando los tipos específicos de los Requisitos de Software.
  • Casos de Prueba: Se contabiliza la cantidad de Casos de Prueba asociados a Requisitos de Usuario y los asociados a Requisitos de Software, separados por estado de cumplimiento. Además se cuentan cuántos de ellos no tienen Tipos de Usuarios asociados.
  • Módulos: Se separan por urgencia, sumando el costo total y contabilizando cuántos de ellos no tienen Requisitos de Software asociados.

Matrices de Trazado

Las matrices de trazado sirven para revisar las relaciones establecidas entre dos tipos de elementos. Ubicando un tipo de elemento en las filas y el otro tipo en las columnas se puede evaluar la cobertura de requisitos, casos de prueba y módulos. De esta manera se puede realizar un análisis de impacto ante alguna modificación de un elemento en particular.

La matriz interactiva permite revisar con un clic el contenido del par de elementos en una intersección. Se destacan con color gris los elementos que no intersecan. Se puede descargar un documento todas las matrices en un archivo PDF, las cuales son particionadas en cuadrantes si sobrepasan el tamaño de la hoja.

  • Las matrices disponibles son las siguientes:
    • Requisitos de Usuario vs. Requisitos de Software
    • Módulos vs. Requisitos de Software
    • Requisitos de Usuario vs. Casos de Prueba
    • Requisitos de Software vs. Casos de Prueba
